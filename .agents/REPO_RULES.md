## Build & Run

The project is **XcodeGen + xcodebuild**, not pure SPM. The `.xcodeproj` is generated from `project.yml` and is gitignored — regenerate after any `project.yml` change.

```bash
brew install xcodegen        # one-time
xcodegen generate            # regenerate SynapseMeetings.xcodeproj
open SynapseMeetings.xcodeproj
```

Local builds use **ad-hoc signing** (`CODE_SIGN_IDENTITY: "-"`, `CODE_SIGNING_ALLOWED: NO`, hardened runtime off) so XcodeGen's defaults work without a Developer ID. The release archive flow flips these at the `xcodebuild` command line — never mutate `project.yml` to do release signing. Full release pipeline (sign + notarize + Sparkle appcast) lives in `.agents/commands/EXPORT-SIGNED-APP.md`.

Version bumps require editing **both** `CFBundleShortVersionString` and `CFBundleVersion` in `project.yml`, then `xcodegen generate`.

## Test

```bash
xcodebuild test -project SynapseMeetings.xcodeproj \
  -scheme SynapseMeetings \
  -destination 'platform=macOS'
```

Single test class:

```bash
xcodebuild test -project SynapseMeetings.xcodeproj \
  -scheme SynapseMeetings \
  -destination 'platform=macOS' \
  -only-testing:SynapseMeetingsTests/AppStatePipelineTests
```

Tests live under `SynapseMeetingsTests/` and use `XCTest`. The test bundle has `BUNDLE_LOADER`/`TEST_HOST` set to the host app, so tests link against the app target and can exercise `AppState`, `RecordingStore`, etc. directly. There is no frontend/backend split — the generic `.agents/commands/TEST-AND-LINT.md` template's frontend/backend wording does not apply to this repo.

There is no Swift linter wired up in this repo. Don't invent one.

## Architecture (the part you can't grep)

Single-window SwiftUI macOS app (deployment target 14.0, Swift 5.10). Apple Silicon strongly recommended — transcription runs on the Neural Engine via Core ML; Intel falls back to CPU and is unusably slow.

**Three layers under `SynapseMeetings/`:**

- `Models/` — `AppState` (the single `@MainActor ObservableObject` injected via `.environmentObject`), `Recording` (Codable, persisted as JSON), `RecordingStore` (filesystem-backed list of recordings).
- `Services/` — wrappers around external concerns: `AudioRecorder`, `TranscriptionService` (FluidAudio / Parakeet), `DiarizationService` (FluidAudio), `AnthropicService`, `GitHubService`, `CalendarService`, `KeychainService`, `UpdaterController` (Sparkle), `GlobalHotkeyService`, `AppMigration`.
- `Views/` — SwiftUI views; all observe `AppState` or take narrow bindings.

**`AppState` is the orchestrator.** It owns one instance of each service and republishes their `objectWillChange` events so SwiftUI re-renders when nested observables update. The pipeline lives in `executePipeline(id:)` in `Models/AppState.swift` — read it to understand the state machine. It walks `RecordingStatus` (`recording` → `transcribing` → `summarizing` → `ready` / `failed`), runs ASR and diarization **in parallel** when both are enabled, then passes either the raw transcript or a speaker-labeled version to Claude. Token-to-speaker alignment is in `alignTokensToSpeakers` and reassembles SentencePiece pieces (`▁` = word boundary).

**Persistence.** Recordings serialize to JSON under `~/Library/Application Support/Synapse Meetings/recordings/`, audio under `…/audio/`. The folder name is a constant on `RecordingStore` — `AppMigration` depends on it for one-time bundle-ID migrations from older installs. The app is **not sandboxed** (`com.apple.security.app-sandbox: false`) so it reads/writes Application Support directly.

**Secrets.** Anthropic API keys and GitHub PATs go through `KeychainService`, never `UserDefaults`. User preferences (model name, prompt overrides, hotkeys, audio device UID, diarization toggle) use `@AppStorage`.

**Live transcription is "replace, not append".** Each chunk re-transcribes the full audio so far, so `liveTranscript` is overwritten — appending would duplicate as Parakeet's context grows. See `handleChunk` in `AppState`.

**Diarization is opt-in** via `diarizationEnabled`. When on, `AppState.init` pre-warms the model so the first recording doesn't pay download/compile cost in-band. First launch downloads ~470MB of Parakeet TDT 0.6B v3 Core ML weights from Hugging Face — `ModelDownloadSheet` is the UI for that.

**Auto-update** ships via Sparkle. Public EdDSA key is in `Info.plist` (`SUPublicEDKey`); the appcast is fetched from `https://raw.githubusercontent.com/dep/synapse-meetings/main/appcast.xml`. The appcast URL must point at the GitHub raw on `main`, and the DMG enclosure URL must match the release asset URL — see EXPORT-SIGNED-APP.

## Commit / PR conventions

Follow `.agents/commands/COMMIT.md` and `OPEN-PR.md`. Recent history shows short conventional-style messages (`feat:`, `chore:`, `bump version to x.y.z`). Version bumps are their own commit.

## Gotchas

- After editing `project.yml`, always `xcodegen generate` before building or running tests, or you'll be testing against stale project settings.
- Don't enable hardened runtime / Developer ID signing in `project.yml` — it breaks local dev. Override at the `xcodebuild` CLI for release only.
- The test target's `TEST_HOST` is the built `.app`; build the app target at least once before running tests from CLI.
- Sparkle's nested XPC helpers (`Installer.xpc`, `Downloader.xpc`, `Autoupdate`, `Updater.app`) occasionally need re-signing after export — EXPORT-SIGNED-APP has the fallback `codesign` commands.
- `AppState` is `@MainActor`. Background `Task`s calling into it must `await` or hop to main.
